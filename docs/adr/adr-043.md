# ADR-043 — Estratégia de Deploy Automático (DEMO) e com Aprovação (PROD) no L2SLedger

> **Status**: Aprovado  
> **Data**: 2026-02-17  
> **Decisores**: Arquiteto, DevOps, Tech Lead  
> **Contexto**: CI/CD, Deploy, Governança

---

## Contexto

O L2SLedger possui três ambientes isolados:
- **DEV**: desenvolvimento local
- **DEMO**: demonstração e testes
- **PROD**: produção com dados reais

Inicialmente, todos os deploys eram **manuais** via `workflow_dispatch`, exigindo trigger humano para qualquer ambiente. Isso garantia controle total, mas criava gargalos operacionais:
- Deploys frequentes em DEMO exigiam intervenção manual repetitiva
- Feedback de integração demorado (aguardar alguém fazer o deploy)
- Risco de esquecer de deployer após merge

Por outro lado, deploy automático **direto em PROD sem aprovação** viola os princípios de governança do projeto (Planejar → Aprovar → Executar) e apresenta riscos inaceitáveis.

---

## Decisão

Implementar **duas estratégias de deploy distintas**:

### 1. Deploy Automático para DEMO

**Trigger**: Automático após `push` na branch `main` ou `release/**`

**Características**:
- Detecta mudanças via `paths-filter` (backend/frontend)
- Usa tag baseada em SHA do commit (`sha-abc1234`)
- Deploy apenas dos serviços alterados
- Health checks obrigatórios pós-deploy
- Falha se health check não passar

**Justificativa**:
- DEMO é ambiente de validação e demonstração (não contém dados críticos)
- Feedback rápido para a equipe
- Testes de integração contínuos
- Reduz overhead operacional

### 2. Deploy Manual com Approval Gate para PROD

**Trigger**: Manual via `workflow_dispatch` no GitHub Actions

**Características**:
- Requer trigger humano explícito
- Especifica tag de versão (`v1.2.3`) manualmente
- Escolhe serviço a deployer (`all`, `backend`, `frontend`)
- **Environment Protection**: GitHub environment `production` com **required reviewers**
- Aguarda aprovação humana antes de executar
- Health checks obrigatórios pós-deploy

**Justificativa**:
- PROD contém dados financeiros reais
- Conformidade com governança (Planejar → Aprovar → Executar)
- Aprovação humana obrigatória antes de mudanças críticas
- Controle total sobre versão deployada

## Arquitetura de Workflows

### Workflow 1: `deploy-demo.yml`

```
push (main/release/**) → Detect Changes → Validate Images → Deploy (auto) → Health Check
```

**Ambiente GitHub**: `demo`  
**Secrets necessários**:
- `DEMO_VM_HOST`
- `DEMO_VM_USER`
- `DEMO_VM_SSH_KEY`
- `DEMO_VM_SSH_PORT` (opcional, padrão: 22)
- `DEMO_VM_DEPLOY_PATH` (opcional, padrão: `/opt/l2sledger`)

### Workflow 2: `deploy.yml` (renomeado)

```
workflow_dispatch → Validate Images → [WAIT FOR APPROVAL] → Deploy → Health Check
```

**Ambiente GitHub**: `production` (com required reviewers configurado)  
**Secrets necessários**:
- `VM_HOST` (ou `PROD_VM_HOST`)
- `VM_USER`
- `VM_SSH_KEY`
- `VM_SSH_PORT`
- `VM_DEPLOY_PATH`
- `PROD_DOMAIN` (opcional, para URL no environment)

---

## Configuração Necessária no GitHub

### Environment `demo`
1. Settings → Environments → New environment → Nome: `demo`
2. (Opcional) Configurar deployment branches: `main`, `release/**`
3. Não configurar required reviewers (deploy automático)

### Environment `production`
1. Settings → Environments → New environment → Nome: `production`
2. **Required reviewers**: Adicionar pelo menos 1 revisor (Tech Lead, DevOps, ou Arquiteto)
3. Configurar deployment branches: apenas `main` via workflow manual
4. (Opcional) Environment secrets específicos de PROD

---

## Fluxo de Deploy Típico

### Para DEMO (Automático)

```
1. Dev faz PR para main
2. PR aprovado e mergeado
3. GitHub Actions detecta push em main
4. Workflow deploy-demo.yml é triggered automaticamente
5. Detecta se backend e/ou frontend mudaram
6. Faz pull das imagens GHCR (tag: sha-abc1234)
7. Deploy automático na VM DEMO
8. Health check
9. ✅ DEMO atualizado
```

### Para PROD (Manual com Aprovação)

```
1. Decisão de deploy para PROD
2. Alguém vai em Actions → Deploy to PROD (Manual + Approval)
3. Preenche image_tag (v1.2.3) e service (all/backend/frontend)
4. Clica "Run workflow"
5. Workflow entra em estado "Waiting for approval"
6. Revisor recebe notificação
7. Revisor analisa mudanças, changelog, testes
8. Revisor aprova (ou rejeita)
9. Se aprovado, deploy é executado
10. Health check
11. ✅ PROD atualizado (ou ❌ rollback se health check falhar)
```

## Segurança

### DEMO
- Deploy automático aceita apenas tags SHA geradas pelo CI
- Imagens devem existir no GHCR (validação prévia)
- Health checks obrigatórios (falha = deploy falha)

### PROD
- Deploy requer aprovação humana explícita
- Aprovador deve ser diferente do autor do código (GitHub enforcement)
- Tag de versão especificada manualmente (controle de versão)
- Rollback documentado em `docs/deployment/10-rollback-recovery.md`

---

## Riscos Mitigados

| Risco | Mitigação |
|-------|-----------|
| Deploy acidental em PROD | Environment protection com required reviewers |
| Versão errada deployada | Tag especificada manualmente + validação de existência da imagem |
| Deploy sem testes | CI obrigatório antes de push em main (testes passam antes de deploy DEMO) |
| Health check falhando em PROD | Workflow falha, notifica, rollback manual documentado |
| Secrets vazados | Secrets gerenciados pelo GitHub Secrets, não commitados |

---

## Consequências

### Positivas
- ✅ DEMO sempre atualizado (feedback rápido)
- ✅ PROD protegido por aprovação humana (governança)
- ✅ Redução de overhead operacional para DEMO
- ✅ Controle total sobre deploys de PROD
- ✅ Auditoria via GitHub Actions logs

### Negativas
- ⚠️ Requer configuração adicional de environments no GitHub
- ⚠️ DEMO pode ficar instável se health checks falharem (mas é isolado)
- ⚠️ Deploy em PROD continua manual (intencional)

---

## Alternativas Consideradas

### 1. Deploy Automático para PROD (Rejeitado)
- **Motivo**: Viola governança e ADR-028 (ambientes isolados)
- **Risco**: Deploy acidental de código não revisado em PROD

### 2. Deploy Manual para DEMO (Rejeitado)
- **Motivo**: Overhead operacional desnecessário
- **Limitação**: DEMO não deve ter mesma criticidade que PROD

### 3. Deploy Automático com Smoke Tests (Futuro?)
- **Consideração**: Deploy automático em PROD após smoke tests em DEMO
- **Status**: Não implementado, pode ser considerado no futuro com mais maturidade
