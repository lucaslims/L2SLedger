# ADR-002 — Autenticação com Firebase (Fluxo Completo) no L2SLedger

## Status
Aprovado

## Contexto

O sistema **L2SLedger** utiliza **Firebase Authentication** como provedor de identidade (IdP) no frontend, enquanto o **Backend (.NET 10)** é responsável por validar o **Firebase ID Token**, aplicar regras de segurança e estabelecer a sessão do usuário via **cookie HttpOnly**.

O Firebase pode exigir **verificação de e-mail**, o que impacta diretamente o fluxo de registro e login.  
Este ADR formaliza o **fluxo completo de autenticação**, desde o cadastro até o uso autenticado do sistema, garantindo segurança, previsibilidade e boa experiência do usuário.

---

## Decisão

Adotamos o seguinte modelo:

### Frontend (SPA React)

Responsável por:

- Registro de usuário
- Login via Firebase Authentication
- Envio e reenvio de e-mail de verificação
- Validação do status `email_verified`
- Obtenção do **Firebase ID Token**
- Envio do token **apenas na autenticação inicial**

O frontend **não persiste tokens** e **não gerencia sessão**.

---

### Backend (.NET 10)

Responsável por:

- Validar o **Firebase ID Token** via Firebase Admin SDK
- Bloquear usuários com e-mail não verificado
- Criar ou recuperar o usuário interno no banco relacional
- Criar o **cookie de sessão HttpOnly**
- Manter o backend **stateless**

O backend **não envia e-mails de verificação**.

---

## Fluxo Completo de Autenticação

### 1. Registro de Usuário

1. Usuário informa e-mail e senha no frontend
2. Frontend chama Firebase Authentication:
   - `createUserWithEmailAndPassword`
3. Firebase cria o usuário
4. Frontend envia e-mail de verificação:
   - `sendEmailVerification`
5. Frontend encerra a sessão local
6. Usuário recebe o e-mail e confirma o endereço

---

### 2. Login

1. Usuário informa e-mail e senha
2. Frontend chama:
   - `signInWithEmailAndPassword`
3. Firebase autentica as credenciais
4. Frontend verifica:
   - `user.emailVerified`

#### 2.1 E-mail NÃO verificado

- Frontend:
  - Reenvia e-mail de verificação
  - Exibe mensagem amigável
  - Bloqueia acesso ao sistema

#### 2.2 E-mail verificado

- Frontend:
  - Obtém o Firebase ID Token:
    - `getIdToken()`
  - Envia o token ao backend

---

### 3. Integração com o Backend

1. Frontend chama:
   ```http
   POST /api/auth/login
   Authorization: Bearer <firebase-id-token>
    ```
2. Backend valida o token via Firebase Admin SDK
3. Backend verifica claims obrigatórias:
    - `email_verified` == true
4. Backend cria ou recupera o usuário interno
5. Backend cria o cookie de sessão:
    - `HttpOnly`
    - `Secure` (em produção)
    - `SameSite=Lax`
6. Backend responde com sucesso
7. Frontend passa a consumir a API autenticada via cookie sem enviar tokens manualmente

---

## Regras de Segurança
- Usuários sem e-mail verificado:
    - Não recebem cookie de sessão
    - Não podem autenticar no backend
    - Devem ser informados via frontend
    - Não podem acessar o sistema

- Firebase ID Tokens:
    - Válidos por 1 hora
    - Não são armazenados no frontend
    - Usados apenas na autenticação inicial
    - Nunca é persistido em localStorage ou sessionStorage

- Cookies de sessão:
    - Criados e gerenciados exclusivamente pelo backend
    - `HttpOnly` para evitar acesso via JavaScript
    - `Secure` em produção para garantir transmissão segura
    - `SameSite=Lax` para mitigar ataques CSRF
    - Enviados automaticamente pelo navegador em chamadas subsequentes

---

## Exemplo de Validação no Backend (.NET 10)

```csharp
if (!decodedToken.Claims.TryGetValue("email_verified", out var verified)
    || verified is not bool isVerified
    || !isVerified)
{
    return Unauthorized("EMAIL_NOT_VERIFIED");
}
// Prosseguir com criação/recuperação do usuário e criação do cookie
```

## Consequências

### Positivas

- Fluxo de autenticação claro e auditável
- Segurança alinhada às boas práticas do Firebase
- Separação clara de responsabilidades
- Frontend simplificado
- Backend stateless

### Negativas

- Usuário depende da verificação de e-mail
- Fluxo inicial ligeiramente mais complexo no frontend

---

## Alternativas Consideradas

1. **Permitir login sem verificação de e-mail**
   - Rejeitado por risco de segurança

2. **Backend enviar e-mails de verificação**
   - Rejeitado por violar responsabilidades do Firebase

3. **Persistir Firebase ID Token no frontend**
   - Rejeitado por risco de vazamento

---

## Conclusão

O fluxo definido garante que **apenas usuários com identidade validada** tenham acesso ao L2SLedger, utilizando o Firebase como IdP e o backend como **boundary de segurança**, com sessões mantidas exclusivamente por **cookies HttpOnly**, de forma segura, simples e escalável.
