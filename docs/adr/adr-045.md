# ADR-045 — Ciclo de Vida do Cookie de Sessão e Renovação Silenciosa no L2SLedger

## Status

Proposto — Implementação pendente de aprovação de fase

---

## Contexto

O **L2SLedger** utiliza um modelo de sessão baseado em **cookie HttpOnly** emitido pelo backend após validação do Firebase ID Token (conforme ADR-001, ADR-002, ADR-004, ADR-005).

### Problema identificado (Bug 5.1 — Fase 6)

A configuração **atual** do cookie de sessão apresenta um **desalinhamento crítico** entre o ciclo de vida do cookie backend e o ciclo de vida do Firebase ID Token:

| Aspecto | Estado atual | Estado desejado |
|---|---|---|
| `Max-Age` do cookie backend | Não documentado / potencialmente longo | 1 hora (alinhado ao Firebase) |
| Refresh automático de sessão | Ausente | Sim — via endpoint dedicado |
| Comportamento ao expirar | Usuário recebe 401 silencioso | Renovação silenciosa automática |
| Firebase ID Token refresh | Gerenciado pelo Firebase SDK | Mantido pelo Firebase SDK |

**Impacto no usuário:** sessões podem ficar ativas além do tempo de validade do Firebase ID Token, ou podem expirar prematuramente sem renovação automática, interrompendo fluxos de trabalho.

---

## Opções Avaliadas

### Opção A — Reduzir TTL do cookie (15–30 min)

**Descrição:** Emitir cookie com `Max-Age` curto sem mecanismo de renovação.

| Prós | Contras |
|---|---|
| Simples de implementar | Usuário precisa re-autenticar com frequência |
| Reduz janela de exposição | UX degradada em sessões longas |
| Zero overhead adicional | Inadequado para uso corporativo |

**Decisão:** Rejeitado — UX inadequada para sistema financeiro de uso diário.

---

### Opção B — Validar + renovar cookie em cada requisição

**Descrição:** Backend valida Firebase ID Token em cada request e emite novo cookie com TTL renovado.

| Prós | Contras |
|---|---|
| Sessão contínua sem intervenção | Overhead de validação em TODA requisição |
| UX fluida | Chamada ao Firebase por request (latência) |
| Simples de implementar | Pode gerar race conditions em requests paralelos |

**Decisão:** Rejeitado — overhead de validação por request é incompatível com NFRs de performance.

---

### Opção C — Cookie curto (1h) + endpoint dedicado de refresh ✅ SELECIONADA

**Descrição:** Cookie com `Max-Age = 1 hora`. Frontend renova silenciosamente o cookie antes da expiração chamando um endpoint dedicado `/api/v1/auth/refresh`, que valida o Firebase ID Token e emite novo cookie.

| Prós | Contras |
|---|---|
| Balanço ideal entre UX e segurança | Complexidade adicional no frontend |
| Overhead apenas no refresh (1x/hora máximo) | Novo endpoint a implementar no backend |
| Cookie alinhado ao ciclo do Firebase ID Token | AuthProvider precisa gerenciar timer |
| Transparente ao usuário | |
| Alinhado a práticas modernas de SPA seguro | |

**Decisão:** Aprovado.

---

## Decisão

Adotar a **Opção C**: cookie de sessão com `Max-Age = 1 hora` e renovação silenciosa automática via endpoint dedicado.

---

## Especificação Técnica

### Backend

**Cookie de sessão:**
```
Max-Age = 3600 (1 hora)
HttpOnly = true
Secure = true (produção)
SameSite = Lax
Path = /
```

**Novo endpoint:**
```
POST /api/v1/auth/refresh
Authorization: Bearer {firebase-id-token}  ← enviado pelo frontend via header
```

**Comportamento:**
1. Frontend envia Firebase ID Token fresco (emitido pelo Firebase SDK)  
2. Backend valida o token com Firebase Admin SDK  
3. Se válido: emite novo cookie (Max-Age reiniciado) → resposta `200 OK`  
4. Se inválido/expirado: resposta `401 Unauthorized` → frontend redireciona para login  

**Regra:** O endpoint de refresh **não** pode ser chamado com cookie expirado — ele exige Firebase ID Token válido no header.

---

### Frontend

**Lógica no `AuthProvider`:**
```typescript
// Executar refresh silencioso 5 minutos antes da expiração do cookie
const COOKIE_TTL_MS = 60 * 60 * 1000;     // 1 hora
const REFRESH_BEFORE_MS = 5 * 60 * 1000;  // 5 minutos antes

// Timer iniciado após login bem-sucedido
setTimeout(async () => {
  const freshToken = await firebaseUser.getIdToken(/* forceRefresh */ true);
  await authService.refresh(freshToken);
}, COOKIE_TTL_MS - REFRESH_BEFORE_MS);
```

**Fluxo de falha:**
- Se o refresh falhar (401): `AuthProvider` dispara logout, redireciona para `/login`  
- Firebase SDK gerencia o refresh do ID Token de forma independente (ciclo de 1 hora)

## Plano de Implementação (Fase Futura)

### Backend (`.NET`)

| Tarefa | Arquivo | Observação |
|---|---|---|
| Alterar `Max-Age` do cookie | `AuthController.cs` → método `Login()` / `SetSessionCookie()` | De atual → `TimeSpan.FromHours(1)` |
| Criar endpoint `POST /refresh` | `AuthController.cs` | Novo action; valida Firebase token e reemite cookie |
| Atualizar contrato público | `docs/adr/adr-022-a.md` | Novo endpoint adiciona breaking em contratos |
| Criar testes | `AuthControllerTests.cs` | Cenários: refresh OK, token expirado, cookie renovado |

### Frontend (React)

| Tarefa | Arquivo | Observação |
|---|---|---|
| Implementar timer de refresh | `app/providers/AuthProvider.tsx` | `setTimeout` gerenciado no contexto de autenticação |
| Criar `authService.refresh()` | `features/auth/services/authService.ts` | Chama `POST /api/v1/auth/refresh` com Firebase token fresco |
| Tratar falha de refresh | `AuthProvider.tsx` | Logout automático + redirect `/login` |
| Atualizar testes | `useLogin.test.ts`, `useLogout.test.ts` | Cobrir novo ciclo de sessão |

## Consequências

### Positivas
- Sessões seguras e alinhadas ao Firebase ID Token
- Renovação transparente ao usuário (zero interrupções)
- Overhead mínimo (1 request por hora de sessão ativa)
- Reduce exposição em caso de comprometimento de cookie

### Negativas
- Complexidade adicional no `AuthProvider`
- Novo endpoint backend a manter
- Edge cases a testar (browser em sleep, múltiplas abas, refresh concurrent)

---

## Alternativas Rejeitadas

* **Opção A (TTL curto sem refresh):** UX inadequada
* **Opção B (refresh por request):** overhead incompatível com NFRs
* **Manter estado atual:** risco de segurança não endereçado

---

## Conclusão

A **Opção C** oferece o melhor equilíbrio entre **segurança** (cookie de curta duração) e **UX** (renovação silenciosa transparente). A implementação é incremental — não rompe contratos existentes — e pode ser realizada em uma fase dedicada de segurança.
