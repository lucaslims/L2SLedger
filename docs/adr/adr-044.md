# ADR-044 — Adição de Tipo (CategoryType) à Entidade de Categoria no L2SLedger

## Status

Aprovado

## Contexto

A entidade `Category` do L2SLedger foi implementada sem uma propriedade de **tipo** que diferencie categorias de receita (`Income`) e despesa (`Expense`). O frontend já implementa esse conceito com `type: 'Income' | 'Expense'` em DTOs, formulários e listagens — porém o backend **ignora** esse campo, resultando em:

* **Desalinhamento de contrato** entre frontend e backend
* **Perda de informação**: o tipo enviado pelo frontend não é persistido
* **Impossibilidade de filtrar** categorias por tipo via API
* **Dados seed** sem classificação de tipo (apenas agrupados por comentários)

Trata-se de um **débito técnico crítico** que compromete a integridade do modelo financeiro.

---

## Decisão

Adicionar a propriedade `CategoryType Type` à entidade `Category`, com impacto em todas as camadas da Clean Architecture:

### Regras de Negócio

1. O tipo (`Income` / `Expense`) é **obrigatório** na criação da categoria
2. O tipo é **imutável** após a criação — não pode ser alterado via `UpdateCategory`
3. **Subcategorias herdam** obrigatoriamente o tipo da categoria pai
4. Quando uma subcategoria é criada com `ParentCategoryId`, o tipo é automaticamente resolvido a partir do pai. O campo `Type` na request de criação é ignorado para subcategorias
5. A API passa a suportar **filtro por tipo** na listagem (`GET /api/v1/categories?type=Income`)

### Modelo de Dados

* Nova coluna `type` (varchar) na tabela `categories`
* Valores possíveis: `"Income"`, `"Expense"`
* Coluna NOT NULL com migração que classifica dados existentes
* Novo índice: `idx_categories_type`

### Enum de Domínio

```csharp
public enum CategoryType
{
    Income = 1,
    Expense = 2
}
```

Armazenado como **string** no banco para legibilidade e compatibilidade direta com o frontend.

---

## Compatibilidade com ADR-022

A adição do campo `Type` ao `CategoryDto` é uma **adição de campo** (backward compatible) — não constitui quebra de contrato conforme ADR-022-a. O campo é **novo**, sem remoção ou alteração de campos existentes.

A adição do campo `Type` ao `CreateCategoryRequest` é uma mudança de contrato de entrada. Porém, o frontend **já envia** esse campo — logo a mudança **alinha o contrato** ao comportamento real.

---

## Impacto por Camada

| Camada | Arquivos Impactados |
|--------|-------------------|
| Domain | `Category.cs`, novo `CategoryType.cs`, `ErrorCodes.cs` |
| Application | DTOs, Use Cases, Validators, Mapper |
| Infrastructure | EF Config, Seeder, Migration, Repository |
| API | Controller (filtro por tipo) |
| Testes | Domain, Application, Contract |

---

## Consequências

### Positivas

* Alinhamento completo entre frontend e backend
* Modelo de dados financeiro mais preciso
* Possibilidade de filtrar categorias por tipo
* Subcategorias com consistência hierárquica de tipo
* Seed data corretamente classificado

### Negativas

* Migration necessária em todos os ambientes
* Todos os testes existentes de Category precisam de atualização

---

## Conclusão

A adição de `CategoryType` corrige um débito técnico crítico, alinhando o modelo de domínio à realidade do sistema financeiro e eliminando a discrepância entre frontend e backend.
