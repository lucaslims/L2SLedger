# ADR-003 — Logout e Revogação de Sessão via Firebase no L2SLedger

## Status

Aprovado

## Contexto

O sistema **L2SLedger** utiliza **Firebase Authentication** como provedor de identidade (IdP) e mantém a sessão do usuário por meio de **cookies HttpOnly** gerenciados exclusivamente pelo **backend (.NET 10)**.

Como o Firebase emite **ID Tokens** e **Refresh Tokens**, é necessário garantir que o processo de **logout** invalide efetivamente a sessão do usuário, evitando:

* Sessões órfãs
* Reutilização de tokens comprometidos
* Inconsistências de segurança entre frontend, backend e Firebase

Este ADR define um modelo claro, seguro e auditável para o processo de logout.

---

## Decisão

O processo de logout no **L2SLedger** será composto por **duas ações obrigatórias e complementares**:

1. **Encerramento da sessão no backend (cookie)**
2. **Revogação server-side da sessão Firebase**

Ambas devem sempre ocorrer juntas.

---

## Fluxo de Logout

### 1. Solicitação de Logout

1. Usuário solicita logout no frontend
2. Frontend chama o backend:

   ```http
   POST /api/auth/logout
   ```

---

### 2. Ações no Backend

Ao receber a solicitação de logout, o backend executa:

1. **Revogação de sessão no Firebase**

   * Utiliza o **Firebase Admin SDK**
   * Executa:

     ```csharp
     RevokeRefreshTokensAsync(uid)
     ```
   * Invalida todos os tokens emitidos antes do logout

2. **Remoção do cookie de sessão**

   * O cookie HttpOnly é removido ou invalidado
   * O browser deixa de enviar credenciais automaticamente

---

### 3. Finalização no Frontend

1. Frontend encerra o estado visual da sessão
2. Usuário é redirecionado para a tela de login

---

## Regras de Segurança

* Logout **sempre** deve incluir revogação server-side
* Logout apenas client-side é **explicitamente proibido**
* O backend é o único responsável por:

  * Revogar sessões Firebase
  * Gerenciar cookies
* O frontend **não tenta invalidar tokens manualmente**

---

## Consequências

### Positivas

* Invalidação real de sessão
* Evita reutilização de tokens comprometidos
* Segurança consistente entre Firebase e backend
* Processo auditável e previsível

### Negativas

* Logout depende de chamada remota ao Firebase
* Pode haver pequena latência adicional

---

## Alternativas Consideradas

* **Logout apenas client-side**

  * Rejeitado por insegurança

* **Revogação periódica automática**

  * Rejeitado por complexidade operacional

* **Apenas remoção de cookie**

  * Rejeitado por não invalidar tokens Firebase

---

## Conclusão

A combinação de **revogação server-side no Firebase** com a **remoção do cookie de sessão** garante um modelo de logout **seguro, coerente e alinhado com práticas corporativas**, sendo adequado para o L2SLedger em ambientes Docker e cloud.
