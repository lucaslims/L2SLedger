# ADR-024 — Arquitetura de Serviços, Guards e UI no L2SLedger

## Status

Aprovado

## Contexto

O **L2SLedger** é um sistema financeiro que exige:

* Segurança por construção
* Previsibilidade de comportamento
* Isolamento de responsabilidades
* Ausência de erros silenciosos

Arquiteturas onde a UI decide permissões, serviços possuem lógica implícita ou guards são opcionais resultam em alto risco operacional, especialmente em sistemas financeiros.

---

## Decisão

Adotar uma arquitetura **guard-first e fail-fast**, com separação rígida entre **UI, serviços e autorização**.

---

## Princípios Arquiteturais

### 1. Interfaces Públicas Imutáveis

* Interfaces expostas à UI são contratos públicos
* Não expõem detalhes internos ou flags técnicas
* São compartilhadas entre UI e guards

---

### 2. Guards como Primeira Linha de Defesa

* Todo método público passa por um guard

* Guards validam:

  * Autenticação
  * Autorização
  * Contexto (período, escopo, permissões)

* Guards lançam erros semânticos (fail-fast)

---

### 3. UI Sem Lógica de Decisão

* UI apenas solicita ações
* UI não decide permissões
* UI não aplica regras de negócio

---

### 4. Serviços Previsíveis

* Serviços possuem comportamento determinístico
* Não retornam null silencioso
* Não possuem lógica condicional baseada em ambiente

---

## Consequências

### Positivas

* Segurança consistente
* Eliminação de regressões
* Facilidade de testes
* Arquitetura previsível para agentes de IA

### Negativas

* Maior rigor no design inicial

---

## Conclusão

A arquitetura de serviços, guards e UI garante que o L2SLedger opere com **segurança estrutural**, previsibilidade e clareza de responsabilidades, essenciais para um sistema financeiro.
