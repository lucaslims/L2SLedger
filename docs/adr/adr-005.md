# ADR-005 — Segurança e Autorização (Firebase-only) no L2SLedger

## Status

Aprovado

## Contexto

O **L2SLedger** requer um modelo de autenticação e autorização **robusto, seguro e simples**, adequado a um **SPA (React)** integrado a um **backend único (.NET 10)**, executando em **containers Docker** na mesma VM e compartilhando a **mesma origem**.

Decisões anteriores já estabeleceram que:

* **Firebase Authentication** é o único provedor de identidade (IdP)
* O backend é o **security boundary** do sistema
* A sessão do usuário é mantida exclusivamente por **cookies HttpOnly**
* O backend é **stateless** e não exposto publicamente

Este ADR consolida e formaliza a estratégia de **segurança e autorização**, deixando explícito o que é permitido, proibido e obrigatório no L2SLedger.

---

## Decisão

Adotar um modelo de segurança baseado nos seguintes pilares:

### 1. Autenticação

* **Firebase Authentication** é o **único emissor de identidade**
* Apenas o **Firebase ID Token** é utilizado
* O refresh de token é responsabilidade exclusiva do **Firebase SDK** no frontend
* O Firebase ID Token é usado **somente na autenticação inicial**

---

### 2. Sessão

* A sessão do usuário é mantida **exclusivamente via cookie HttpOnly**
* O cookie é criado e gerenciado apenas pelo **backend**
* O frontend **não acessa, armazena ou manipula** tokens ou cookies
* O backend permanece **stateless**

---

### 3. Autorização

* A autorização é **sempre decidida no backend**
* O backend resolve:

  * Identidade do usuário
  * Contexto de autorização
  * Permissões e restrições
* O frontend **nunca decide autorização**, apenas reflete estados permitidos

---

### 4. Boundary de Segurança

* O backend atua como **único boundary de segurança** do sistema
* Nenhum outro componente gerencia:

  * Sessão
  * Cookies
  * Tokens
* Não existe BFF ou camada intermediária adicional

---

## O que é Explicitamente Proibido

* Emissão de **JWT próprio** pela aplicação
* Uso de **refresh tokens próprios**
* Implementação de `TokenService` customizado
* Armazenamento de tokens no frontend:

  * `localStorage`
  * `sessionStorage`
  * `IndexedDB`
* Decisão de autorização no frontend

---

## Fail-Fast

O sistema deve falhar de forma **rápida e explícita**:

* **401 Unauthorized**

  * Token ausente
  * Token inválido
  * Sessão inexistente

* **403 Forbidden**

  * Usuário autenticado sem permissão

* Erros de segurança **nunca são silenciosos**

* Mensagens de erro devem ser **claras e sem vazar informações sensíveis**

---

## Consequências

### Positivas

* Segurança delegada a um IdP maduro
* Redução significativa da superfície de ataque
* Arquitetura simples, clara e auditável
* Eliminação de duplicação de responsabilidade
* Forte alinhamento com boas práticas modernas de SPA + API

### Negativas

* Dependência direta do Firebase Authentication
* Toda autenticação depende da disponibilidade do IdP

---

## Alternativas Consideradas

* **JWT próprio emitido pelo backend**

  * Rejeitado por duplicar responsabilidades e aumentar complexidade

* **Sessão stateful no backend**

  * Rejeitado por impactar escalabilidade

* **Autorização no frontend**

  * Rejeitado por insegurança

---

## Conclusão

O modelo **Firebase-only + backend como boundary de segurança + cookies HttpOnly** fornece ao L2SLedger uma base sólida de segurança, simplicidade operacional e clareza arquitetural, adequada para ambientes corporativos executando em Docker e cloud.
