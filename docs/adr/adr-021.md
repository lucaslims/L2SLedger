# ADR-021 — Modelo de Erros Semântico e Fail-Fast no L2SLedger

## Status

Aprovado

## Contexto

O **L2SLedger** não utiliza BFF nem microsserviços; porém, por se tratar de um sistema financeiro, exige **previsibilidade absoluta de falhas**, rastreabilidade e comunicação clara de erros para o frontend.

Erros inconsistentes ou silenciosos geram:

* Ambiguidade no frontend
* Dificuldade de auditoria
* Baixa confiabilidade operacional

---

## Decisão

Adotar um **modelo único e semântico de erros**, com aplicação rigorosa do princípio **Fail-Fast**, válido para toda a API do L2SLedger.

---

## Estrutura Padrão de Erro

Todo erro retornado pela API deve seguir a estrutura:

```json
{
  "error": {
    "code": "string",
    "message": "string",
    "details": "string | null",
    "timestamp": "ISO-8601",
    "traceId": "string"
  }
}
```

---

## Princípios Fail-Fast

* Nenhum erro retorna HTTP 200
* Nenhum erro é silencioso
* Exceções interrompem imediatamente o fluxo
* Erros são comunicados no primeiro ponto possível

---

## Categorias de Erro

* **AUTH** — autenticação e autorização
* **VAL** — validação de dados
* **FIN** — regras financeiras e domínio
* **INT** — integrações externas
* **SYS** — falhas internas

Códigos são estáveis e determinísticos.

---

## Comportamento do Frontend

* Frontend reage ao campo `error.code`
* Mensagens exibidas conforme UX definida
* `traceId` pode ser apresentado em erros críticos

---

## Observabilidade

* Todo erro gera log estruturado
* `traceId` é propagado para logs
* Nunca logar dados sensíveis

---

## Consequências

### Positivas

* UX previsível
* Debug facilitado
* Base sólida para auditoria e observabilidade

### Negativas

* Disciplina obrigatória na implementação

---

## Conclusão

O modelo de erros semântico e fail-fast garante que o L2SLedger opere com **previsibilidade, rastreabilidade e segurança**, requisitos essenciais para sistemas financeiros.
