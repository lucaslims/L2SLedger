# ADR-006 — Observabilidade (Logs, Métricas e Tracing) no L2SLedger

## Status

Aprovado

## Contexto

O **L2SLedger** é executado em uma arquitetura baseada em **VMs na Oracle Cloud Infrastructure (OCI)**, utilizando **containers Docker**, com um **frontend SPA (React)** e um **backend único (.NET 10)** compartilhando a mesma VM e origem.

Diferentemente de ambientes totalmente gerenciados, toda a responsabilidade de **observabilidade** recai sobre a própria aplicação e a VM hospedeira.

Neste contexto, é necessário definir uma estratégia clara, padronizada e obrigatória de observabilidade para garantir:

* Visibilidade operacional
* Diagnóstico de falhas
* Auditoria de eventos críticos
* Suporte a fluxos sensíveis (importação, conciliação financeira, autenticação)

---

## Decisão

### 1. Princípios Gerais

* Observabilidade é **obrigatória** no backend
* Implementação **padronizada e consistente**
* Serviços permanecem **stateless**
* Correlação de requisições via **Correlation ID / Trace ID**
* Nenhuma dependência obrigatória de serviços gerenciados de cloud

---

### 2. Logs

#### Estratégia

* Logs **estruturados em JSON**
* Escritos em **stdout/stderr** dos containers
* Coleta realizada por agente da VM ou stack externa (ex: ELK, Loki)

#### Regras

* É **proibido** logar:

  * Firebase ID Tokens
  * Cookies
  * Credenciais
  * Dados sensíveis (PII, dados financeiros detalhados)

* Toda requisição ao backend deve registrar:

  * Timestamp
  * Correlation ID
  * Endpoint
  * Método HTTP
  * Status Code
  * Tempo de execução

* Eventos críticos devem ser explicitamente logados:

  * Login
  * Logout
  * Importação de dados
  * Erros de validação
  * Falhas de conciliação

---

### 3. Métricas

#### Estratégia

* Métricas expostas via endpoint interno (ex: `/metrics`)
* Coleta realizada por ferramenta externa (ex: Prometheus)

#### Métricas mínimas

* Total de requisições
* Latência (p50, p95, p99)
* Taxa de erro (4xx / 5xx)
* Uso de CPU e memória por container
* Conexões ativas com o banco de dados

---

### 4. Tracing

#### Estratégia

* Tracing distribuído implementado na aplicação
* Propagação de Trace ID entre:

  * Frontend → Backend
  * Backend → Banco de dados
  * Backend → Serviços externos (Firebase)

#### Regras

* Cada requisição pertence a um único trace
* Spans obrigatórios para:

  * Entrada HTTP
  * Acesso a banco de dados
  * Chamada a serviços externos

---

### 5. Armazenamento e Retenção

* Logs devem ser armazenados fora dos containers
* Política de retenção definida por ambiente (DEV / HML / PROD)
* Dados de observabilidade **não devem** ser persistidos nos volumes de dados do banco

---

## Consequências

### Positivas

* Visibilidade completa da aplicação
* Facilidade de troubleshooting em produção
* Base sólida para auditoria e compliance
* Independência de vendor cloud

### Negativas

* Maior esforço operacional
* Necessidade de manutenção da stack de observabilidade

---

## Alternativas Consideradas

* Observabilidade dependente de cloud provider
* Logs não estruturados
* Ausência de métricas

Todas rejeitadas por comprometerem visibilidade e operação segura.

---

## Conclusão

A adoção de uma estratégia de observabilidade **auto-gerenciada, padronizada e obrigatória** garante que o L2SLedger opere de forma previsível, auditável e confiável em ambiente Docker e OCI, sustentando fluxos críticos e evolução futura do sistema.
