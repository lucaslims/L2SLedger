# ADR-004 — Segurança de Cookies, CSRF e SameSite no L2SLedger

## Status
Aprovado

## Contexto

O sistema **L2SLedger** é composto por um **SPA (React)** e um **Backend (.NET 10)** executando na **mesma VM**, em containers Docker, compartilhando a **mesma origem** (domínio/host).

Apenas o **frontend é exposto publicamente**, enquanto o backend não possui exposição direta à internet. A autenticação é baseada em **Firebase Authentication**, com validação inicial do Firebase ID Token no backend.

Era necessário definir de forma explícita:

- Estratégia de uso de cookies
- Política de SameSite
- Mitigação de CSRF
- Responsabilidades de segurança do backend
- Proibição de práticas inseguras (ex: LocalStorage)

---

## Decisão

1. **O backend é o único responsável pela gestão de cookies**
   - O frontend nunca cria, altera ou lê cookies de autenticação
   - O backend atua como *security boundary* do sistema

2. **Cookies de autenticação seguem obrigatoriamente o padrão**
   - `HttpOnly`
   - `Secure` (obrigatório em produção)
   - `SameSite=Lax`
   - `Path=/`

3. **Política SameSite**
   - `SameSite=Lax` é o **padrão oficial e obrigatório**
   - `SameSite=Strict` **não será utilizado**
   - `SameSite=None` é **explicitamente proibido**

4. **Tokens nunca são armazenados no frontend**
   - Proibido uso de:
     - `localStorage`
     - `sessionStorage`
     - `IndexedDB`
   - O Firebase ID Token é usado **apenas na troca inicial de autenticação**

5. **Mitigação de CSRF**
   - A aplicação opera em **mesma origem**, reduzindo riscos inerentes
   - Cookies `HttpOnly + SameSite=Lax` fornecem proteção base adequada
   - O backend deve validar:
     - Método HTTP
     - Origem da requisição quando aplicável
   - Tokens CSRF adicionais **não são necessários no MVP**, mas podem ser introduzidos futuramente se o modelo evoluir

6. **Backend não é exposto publicamente**
   - Não há CORS aberto para terceiros
   - Comunicação frontend → backend ocorre via mesma origem
   - Containers se comunicam por Docker Network interna

---

## Justificativa Técnica

- O uso de `SameSite=Lax` é o **melhor equilíbrio** entre:
  - Segurança
  - Compatibilidade com SPA
  - Simplicidade operacional
- `SameSite=Strict` foi descartado por:
  - Quebrar fluxos legítimos de navegação e autenticação
- `SameSite=None` foi descartado por:
  - Não ser necessário em ambiente de mesma origem
  - Aumentar a superfície de ataque
- Cookies `HttpOnly` eliminam vetores comuns de XSS
- Backend isolado reduz drasticamente riscos de exploração direta

---

## Consequências

### Positivas
- Arquitetura simples e segura
- Redução significativa da superfície de ataque
- Eliminação de riscos comuns de XSS e CSRF
- Clareza absoluta de responsabilidades
- Facilidade de auditoria e manutenção

### Negativas
- Forte dependência da correta implementação do backend
- Qualquer falha no backend impacta diretamente autenticação

---

## Alternativas Consideradas

- **SameSite=Strict**
  - Rejeitado por quebrar fluxos legítimos de SPA
- **SameSite=None**
  - Rejeitado por ser desnecessário e menos seguro
- **Tokens em LocalStorage**
  - Rejeitado por alto risco de XSS
- **Cookies manipulados no frontend**
  - Rejeitado por quebra de boundary de segurança

---

## Conclusão

O modelo adotado — **cookies HttpOnly + Secure + SameSite=Lax**, backend isolado e Firebase como IdP — fornece um nível de segurança **adequado para ambientes corporativos**, mantendo simplicidade, clareza arquitetural e alinhamento com boas práticas modernas de SPA + API.
