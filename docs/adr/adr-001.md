# ADR-001 — Autenticação Centralizada via Firebase com Cookies no L2SLedger

## Status
Aprovado

## Contexto

O sistema **L2SLedger** utiliza **Firebase Authentication** como provedor de identidade (IdP).  
A aplicação é composta por:

- **Frontend**: SPA em React
- **Backend**: API .NET 10 stateless
- Ambos executando na **mesma VM**, em containers Docker, compartilhando a **mesma origem**
- Apenas o frontend é exposto publicamente

Era necessário definir de forma clara e inequívoca:

- Quem é responsável pela identidade
- Onde ocorre a validação do token
- Como a sessão é mantida
- Onde cookies são criados e gerenciados
- Como manter segurança sem introduzir complexidade desnecessária

A decisão precisava preservar:
- Clean Architecture
- Segurança
- Simplicidade operacional
- Clareza de responsabilidades

---

## Decisão

1. **Firebase Authentication é o único emissor de identidade**
   - Firebase emite:
     - **ID Token (JWT)**
     - **Refresh Token** (gerenciado internamente pelo SDK)
   - O sistema **não emite tokens próprios de identidade**

2. **O Firebase ID Token é usado apenas na autenticação inicial**
   - Utilizado exclusivamente no momento do login
   - Nunca persistido no frontend
   - Nunca reutilizado após a troca inicial

3. **O backend valida o Firebase ID Token**
   - Validação feita via **Firebase Admin SDK**
   - Claims obrigatórias (ex: `email_verified`) são verificadas
   - Tokens inválidos ou expirados são rejeitados imediatamente

4. **O backend cria e gerencia a sessão via cookie**
   - Cookie de autenticação criado após validação do token
   - Cookie segue obrigatoriamente o padrão:
     - `HttpOnly`
     - `Secure` (em produção)
     - `SameSite=Lax`
   - O frontend **não tem acesso ao cookie**

5. **Nenhuma API emite token próprio**
   - Não existe:
     - Access Token da aplicação
     - Refresh Token da aplicação
   - O backend permanece **stateless**

6. **Firebase é apenas Identity Provider**
   - Dados de domínio não residem no Firebase
   - Informações como:
     - Usuário interno
     - Permissões
     - Relacionamentos
   - Vivem exclusivamente no banco relacional

---

## Justificativa Técnica

- Firebase fornece:
  - Infraestrutura segura
  - Gestão de identidade madura
  - Redução de complexidade
- Cookies `HttpOnly`:
  - Eliminam vetores de XSS
  - Simplificam o frontend
- Backend único como *security boundary*:
  - Facilita auditoria
  - Reduz acoplamento
- Ausência de tokens próprios:
  - Evita duplicação de estado
  - Evita problemas de rotação e expiração

---

## Consequências

### Positivas
- Arquitetura simples e clara
- Redução da superfície de ataque
- Backend completamente stateless
- Separação limpa entre identidade e domínio
- Facilidade de manutenção e auditoria

### Negativas
- Dependência direta do Firebase Authentication
- Todas as requisições autenticadas dependem de cookies

---

## Alternativas Consideradas

- **Emissão de JWT próprio pelo backend**
  - Rejeitado por duplicar responsabilidades e aumentar complexidade
- **Persistência do Firebase ID Token no frontend**
  - Rejeitado por risco de segurança
- **Autenticação baseada em LocalStorage**
  - Rejeitado por vulnerabilidade a XSS
- **Sessão stateful no backend**
  - Rejeitado por impacto em escalabilidade

---

## Conclusão

A decisão de utilizar **Firebase Authentication como único provedor de identidade**, combinada com **cookies HttpOnly gerenciados exclusivamente pelo backend**, fornece um modelo de autenticação **seguro, simples e alinhado com práticas modernas** para aplicações SPA executando em ambiente Docker e cloud.
